# 合同工作共享协议

本协议适用于所有合同起草、合同审查、合同续约提醒、合同问答、版本比对和最终版归档任务。事项路径和系统记录隔离以 `matter-workspace-protocol.md` 为准；本协议只补充合同层面的编号、记录、续约、版本和输出规则。合同偏好学习另见 `contract-preference-learning-protocol.md`。

## 0. 合同审查立场硬闸门

所有合同审查、合同审核、合同红线修订、合同风险评估、合同问题清单、合同飞书正式交付和合同 Word 修订稿任务，必须先由用户明确确认审查立场，才能开始任何实质审查。

### 0.1 必须先问

用户未明确说明审查立场时，必须先问：

```text
这份合同按哪一方立场审？甲方、乙方，还是中立审查？
```

在用户确认前，不得进行以下任何动作：

- 不得输出初步审核意见、问题清单、风险点、修改建议或批注建议。
- 不得生成飞书文档、思维导图、流程图、Word 修订稿、清洁版或任何正式/工作交付物。
- 不得把合同当作“中立初筛”继续审。
- 不得根据文件名、律所名称、我方姓名、合同模板来源、当前事项或上下文自行推断立场。

### 0.2 允许的确认值

- `甲方`
- `乙方`
- 具体合同角色，例如 `买方`、`卖方`、`出租方`、`承租方`、`委托人`、`受托人`
- `中立审查`
- 用户明确描述的其他一方或多方立场

### 0.3 记录要求

确认后必须写入或同步到合同记录：

- `合同概览.md` 的 `代理立场`
- `用户确认记录.md` 或等效记录中的 `审查立场确认`
- 如生成红线稿，`redline-plan.json` 的 `meta.party_role` 和 `meta.party_role_confirmed=true`

未确认时，相关字段不得写成模型推断值；只能写 `待用户确认`，并停止审查流程。

## 一、合同编号

合同编号必须方便客户口头询问，优先使用“客户/项目简称 + 合同关键词 + 标的或序号”。

推荐格式：

```text
客户简称-合同关键词-标的或序号
```

示例：

```text
刘明-委托租赁-3F特29
刘明-委托租赁-3F特30
威宝恒-供货合同-01
张学智-和解协议-01
某公司-SaaS服务-01
```

规则：

1. 合同原文已有合同编号、订单号、项目号的，记录为“原合同编号”，但不替代客户可读编号。
2. 同一客户多份同类合同，优先用标的物、房号、项目名区分；无法区分时用 `01`、`02`。
3. 用户常用说法写入别名，如“特29那份”“星火租赁合同”“第一份租赁合同”。
4. 后续合同问答必须先查 `合同索引.md`，不得靠模型记忆判断用户说的是哪份合同。
5. 编号冲突时，在原编号后追加短标识，如 `刘明-委托租赁-3F特29-补充协议`。

## 二、合同路径

新合同按“双路径”建档，不批量改造历史合同。

### 2.1 业务文件区

给用户看的合同原件、工作版本、交付文件和最终版放入：

```text
【自定义工作目录】/
  合同/
    <客户或项目>/
      <合同编号-合同简称>/
        01-客户材料/
        02-工作版本/
        03-交付文件/
        04-最终版/
```

### 2.2 系统记录区

系统内部需要读取、复查、溯源、问答和偏好学习使用的记录放入：

```text
【自定义工作目录】/
  _系统记录/
    合同/
      合同索引.md
      续约提醒总表.md
      <客户或项目>/
        <合同编号-合同简称>/
          合同概览.md
          业务摘要.md
          审查问题清单.md
          谈判立场与让步记录.md
          续约提醒.md
          版本记录.md
          修订历史.md
          版本对比记录.md
          最终版确认记录.md
          归档记录.md
          来源验证记录.md
          OCR校正记录.md
          OCR文本/
```

业务文件区与系统记录区必须在 `合同索引.md` 中同时记录。

## 三、合同索引

`合同索引.md` 至少记录：

```text
| 合同编号 | 别名 | 合同名称 | 合同类型 | 我方/客户 | 相对方 | 原合同编号 | 当前状态 | 最近处理日期 | 业务文件路径 | 系统记录路径 |
```

当前状态建议使用：`待审查`、`审查中`、`待客户确认`、`起草中`、`谈判中`、`待定稿`、`已定稿`、`待签署`、`已签署`、`履行中`、`已终止`、`已归档`。

## 五、业务摘要

`业务摘要.md` 用于快速问答，不写成长篇法律意见：

```text

- 合同编号：

- 当前状态：
- 下一步动作：
```

## 六、版本与最终版归档

本机制同时适用于合同审查和合同起草。只有用户明确表示“这是最终版”“定稿”“归档”“签署版”时，才能标记为最终版；否则只能记录为工作版本，不得替用户猜定稿。

### 6.1 版本文件夹

合同版本文件按性质放入：

```text
业务文件区：
  01-客户材料/   # 首次收到的合同、附件、补充协议、对方模板等
  02-工作版本/   # AI草案、客户修改稿、对方反馈稿、谈判中版本
  03-交付文件/   # 飞书导出的审查报告、给客户看的摘要或交付材料
  04-最终版/     # 用户明确确认的最终版、签署版、归档版
```

不得覆盖原文件；系统记录区只记录文件路径、版本关系和差异，不存放给用户看的正式文件。

### 6.2 版本编号

工作版本用简单编号：

```text
V1 初稿
V2 客户修订稿
V3 对方反馈稿
V4 定稿
```

补充协议或多文件修订用文件性质命名：

```text
主合同
补充协议一
补充协议二
续签协议
解除协议
Addendum
Amendment 1
Amendment 2
```

### 6.3 版本记录

每次起草、审核、修改或收到新版本，都更新 `版本记录.md`：

```text
| 版本 | 日期 | 来源 | 文件 | 基于版本 | 本轮目的 | 状态 |
|---|---|---|---|---|---|---|
| V1 | 2026-xx-xx | AI起草/客户提供/对方反馈/【律师姓名】确认 | 文件路径 | - | 初稿/审查/修改/定稿 | 工作版/最终版/已替换 |
```

并追加本轮说明：

```text
## V2 客户修订稿
- 生成/接收时间：
- 基于版本：
- 来源/用户指令：
- 本轮修改摘要：
- 影响条款：
- 是否改变核心交易条件：
- 是否改变业务风险：
- 是否改变续约/期限提醒：
- 输出/保存文件路径：
```

### 6.4 修订历史

多份合同版本、补充协议、Addendum、Amendment 等必须先排序，再写入 `修订历史.md`：

排序依据优先级：

1. 文件标题中的修订编号，如 Amendment 1、补充协议一
2. 签署日期或生效日期
3. 文内引用的原合同或前序补充协议
4. 文件名日期

排序冲突或无法判断时，必须提示用户确认，不得自行脑补。

`修订历史.md` 格式：

```text
## 整体历史
| 顺序 | 文件 | 签署/生效日期 | 文件性质 | 修改对象 | 当前效力 | 来源 |
|---|---|---|---|---|---|---|

## 每次修订改了什么
### V2 / Amendment 1 / 补充协议一
- 修改条款：
- 新增条款：
- 删除/替换条款：
- 对金额/期限/付款/违约/管辖/续约的影响：
- 是否影响当前审查或起草结论：
- 来源：
```

### 6.5 版本对比记录

收到修订稿、对方反馈稿、最终版、补充协议或 Amendment/Addendum 时，必须写入 `版本对比记录.md`：

```text
# 版本对比记录

## 本次对比
- 新版本：
- 对比基准：
- 对比时间：
- 对比目的：审核后最终版确认 / 起草后修订 / 对方反馈 / 签署版归档 / 补充协议排序

## 差异摘要
- 修改条款：
- 新增条款：
- 删除条款：
- 替换条款：
- 格式或表述调整：

## 关键业务条件变化
- 金额/价格：
- 付款节点：
- 履行期限：
- 交付/验收：
- 违约责任：
- 解除条件：
- 管辖/仲裁：
- 续约/到期提醒：

## 判断
- 是否改变原审查/起草结论：
- 是否新增风险：
- 是否仍需客户确认：
- 是否可进入最终版归档：
```

### 6.6 最终版确认

用户提供最终版、定稿、签署版或要求归档时，必须读取最终版并与基准版本比对。基准版本通常为上一工作版本、被审查版本或 AI 起草版本；如无法判断基准版本，先询问用户。

`最终版确认记录.md` 格式：

```text
# 最终版确认记录

- 最终版文件：
- 确认时间：
- 用户确认方式：
- 对比基准：
- 是否采纳主要审查/起草意见：
- 未采纳事项：
- 新增修改：
- 新增风险：
- 仍需客户确认事项：
- 是否改变续约/期限提醒：
- 当前状态：已定稿 / 待签署 / 已签署 / 已归档
```

### 6.7 归档记录

最终版确认后，更新 `归档记录.md`：

```text
# 归档记录

- 合同编号：
- 最终版文件：
- 最终版所在目录：
- 关联工作版本：
- 关联原始材料：
- 归档时间：
- 归档状态：
- 后续动作：
```

同时更新 `合同索引.md` 的当前状态。

## 七、审查问题格式

合同审查生成飞书交付物前，必须同步提示用户以下输出说明，并写入 `来源验证记录.md`：

```markdown
## 一、输出说明
- 已读取材料：
- 已核验内容：
- 未核验/存疑内容：
- 法规案例检索状态：
- 需要客户判断事项：
```

该输出说明用于暴露读取、核验、缺口和客户判断事项，不进入飞书正文。

每个问题按以下结构输出并同步写入 `审查问题清单.md`：

```text
### 问题 001｜付款条件不清
- 条款位置：
- 原文摘录：
- 问题概述：
- 法律风险：
- 业务风险：
- 处理建议：必须修改 / 建议修改 / 需客户确认 / 可优化
- 建议修改文本：
- 对方不同意时：
- 可接受底线/需客户判断：
- 可复制为批注的短文本：
- 来源：
```

飞书正式交付只输出上述问题卡片列表和业务流程图；合同概览、业务摘要、综合审核意见、来源验证、续约提醒等保存为本地 Markdown。

“对方不同意时”必须给出谈判落点，通常从以下选项中选择：

- 坚持修改，不建议让步
- 可改为折中条款
- 可接受但需降低价格、增加担保或缩短账期
- 可接受但必须由客户确认商业风险
- 不建议签署

## 八、续约提醒

凡合同存在期限、自动续约、提前通知解除/不续约条款，必须抽取续约提醒字段：

```text
| 合同编号 | 到期日 | 自动续约方式 | 提前通知天数 | 最晚发通知日 | 通知方式 | 通知地址/邮箱 | 是否需要工作日顺延 | 状态 | 来源条款 |
```

状态建议使用：`信息缺失`、`未设置`、`已设置飞书提醒`、`已通知`、`已续约`、`已终止`。

规则：

1. `最晚发通知日` 必须根据合同到期日、提前通知天数、工作日顺延规则计算；涉及日期计算时必须使用系统命令或脚本辅助，不得心算。
2. 合同未写清到期日、提前通知天数、通知方式或通知地址时，标为 `信息缺失`，提示用户补充，不创建日历。
3. 法定节假日顺延不能凭模型记忆判断；需要检索或由用户确认。
4. 续约提醒写入系统记录区单合同 `续约提醒.md` 和 `_系统记录/合同/续约提醒总表.md`。
5. 创建飞书日历属于写入操作，必须先向用户确认。

用户确认后，用飞书日历创建提醒：

```bash
lark-cli calendar +create \
  --as user \
  --summary "合同续约提醒｜<合同编号>" \
  --start "<最晚发通知日>T09:00:00+08:00" \
  --end "<最晚发通知日>T09:30:00+08:00" \
  --description "合同编号：<合同编号>
到期日：<到期日>
自动续约方式：<自动续约方式>
提前通知天数：<提前通知天数>
最晚发通知日：<最晚发通知日>
通知方式：<通知方式>
通知地址/邮箱：<通知地址/邮箱>
是否需要工作日顺延：<是否需要工作日顺延>
来源条款：<来源条款>"
```

默认提醒时间为最晚发通知日当天上午 9:00 至 9:30。用户另有习惯时，以用户指定为准。

## 九、合同问答

用户询问某份合同时：

1. 先读取 `_系统记录/当前事项.md`，再读取 `_系统记录/合同/合同索引.md`。
2. 根据合同编号、别名、合同名称、标的物匹配合同。
3. 再读取系统记录区该合同的 `合同概览.md`、`业务摘要.md`、必要时读取 `审查问题清单.md`，并按索引读取业务文件区的原始材料。
4. 如果无法唯一匹配，列出候选合同让用户选择。
5. 回答必须说明来源，不得把记忆当成合同内容。

## 十、合同偏好学习

合同审查、合同起草、版本比对和最终版归档完成后，如出现用户接受、拒绝、让步或保留某类条款的明确结果，按 `contract-preference-learning-protocol.md` 记录可学习样本。

触发偏好提案时，只能生成提案并询问用户确认；不得自动修改合同专项审查规则，不得把历史接受包装成“无风险”。
