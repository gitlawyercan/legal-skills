# 审查交付规范

本文件规定合同审查 Skill 的交付物格式。正式交付以飞书文档为准，但飞书正文只输出问题卡片列表；业务流程图作为飞书交付的必选组成部分。其他内容保存为本地 Markdown 记录。

## 一、飞书正式交付

飞书文档必须包含以下两类内容，缺一不可：

1. 问题卡片列表
2. 业务流程图（渲染图片 + Mermaid 源码；如合同信息不足以生成流程图，应在本地来源验证记录中说明）

业务流程图是合同审查飞书正式交付物的强制交付项。不得只交付问题卡片列表；不得用未渲染的 Mermaid 代码替代流程图图片。合同信息不足以生成流程图时，飞书正式交付状态为阻塞或未完成，必须提示用户补充材料或确认缩小交付范围。

生成飞书文档前，必须在对话中向用户同步提示以下输出说明，并写入本地 `来源验证记录.md`：

```markdown
## 一、输出说明
- 已读取材料：
- 已核验内容：
- 未核验/存疑内容：
- 法规案例检索状态：
- 需要客户判断事项：
```

该输出说明不进入飞书正文。

创建命令：

```bash
lark-cli docs +create --as user --markdown "文档内容"
```

创建或更新飞书文档后，必须执行交付检查：

1. 记录飞书文档 Token、链接或可定位标识。
2. 检查飞书正文包含问题卡片列表和业务流程图。
3. 检查业务流程图图片已生成、可打开、已嵌入或随飞书文档交付；Mermaid 源码已保存为本地记录。
4. 检查飞书正文的问题卡片与本地 `审查问题清单.md` 一致。
5. 将飞书链接、流程图文件路径和检查结果写入 `来源验证记录.md` 或 `飞书同步记录.md`。

## 二、问题卡片格式

每个问题必须且只按以下结构输出：

```markdown
### 问题 001｜付款条件不清
- 条款位置：
- 原文摘录：
- 问题概述：
- 法律风险：
- 业务风险：
- 处理建议：必须修改 / 建议修改 / 需客户确认 / 可优化
- 建议修改文本：
- 对方不同意时：
- 可接受底线/需客户判断：
- 可复制为批注的短文本：
- 来源：
```

要求：

1. 不标注“风险等级”。
2. “处理建议”只使用：必须修改、建议修改、需客户确认、可优化。
3. “建议修改文本”必须给出可直接替换或新增的完整条款；新增条款必须说明插入位置。
4. “可复制为批注的短文本”应简短，可直接贴到合同批注中，不重复长篇分析。
5. “来源”必须明确是合同原文、用户口述、文件材料、企业核验、法规/案例检索、网页搜索还是模型分析。

## 三、本地 Markdown 记录

以下内容不进入飞书正文，保存到合同事项的系统记录区：

## 四、业务流程图

从合同文本提取关键时间点、金额、交付节点、验收节点、违约条件，生成 Mermaid flowchart。

规则：

1. 流程图必须严格来自合同文字，不臆测。
2. 合同信息不足以画图时，不强行补全；在本地来源验证记录中说明缺口。
3. 必须渲染为图片并检查可读性；存在节点拥挤、文字重叠、断字、箭头方向不清时，必须拆图或改图型后重新渲染。
4. 可通过以下命令渲染：

```bash
mmdc -i flow.mmd -o flow.png -t neutral
```

## 五、修订模式交付

用户明确要求“修订模式”“红线稿”“用 Word 修订格式审核/修改”“生成合同审核修订稿”时，合同审查可以在问题卡片之外，以原合同为只读来源另行生成修订模式 Word 审核红线稿。该红线稿不属于修改合同原文，而是一个新建的合同审核成果。修订模式交付必须遵守法律工作总控的 `修订稿正式交付强制链路`：

1. 原合同文件只作为读取和比对来源，必须保留不变；修订稿另存为新的 Word 文档。
2. 读取 `references/修订策略.md` 和 `references/redline-plan-protocol.md`，生成 `redline-plan.json`，逐项记录条款位置、原文、修改后文本、操作类型和命中状态。
3. 在 `redline-plan.json` 中记录审查人信息、客户名称、审查立场、审查口径、审查时限/上线期、原合同路径、正式审查意见书或飞书审查报告路径；缺失项写 `未提及/待补充`。
4. 优先运行 `scripts/redline/apply_redline_plan.py` 生成真实 DOCX 批注和修订痕迹。
5. 检查 DOCX 真实修订结构：`w:trackRevisions`、`w:ins`、`w:del`。
6. 运行 `scripts/redline/qa_redline.py` 或等效结构 QA，检查批注、关系文件和修订数量。
7. 检查接受修订后的临时清洁文本，确认关键条款、编号、交叉引用和签署页正确。
8. 渲染红线稿逐页检查；如同时交付清洁版，清洁版也必须渲染检查。

修订模式不自动替代飞书正式交付物。除非用户明确只要修订稿，否则合同审查正式交付仍应包含飞书问题卡片列表和业务流程图。

## 六、示例

```markdown
### 问题 001｜验收标准不明确
- 条款位置：第5条第2款
- 原文摘录：“甲方验收合格后支付尾款。”
- 问题概述：合同未约定验收标准、验收期限和不合格处理方式。
- 法律风险：发生质量争议时，双方难以判断是否达到付款条件，可能导致付款和违约责任争议。
- 业务风险：我方可能已经完成主要工作，但因验收口径不明导致尾款长期无法回收。
- 处理建议：必须修改
- 建议修改文本：建议将第5条第2款修改为：“甲方应在收到乙方提交的交付成果及完整验收资料后5个工作日内完成验收。验收标准以本合同附件一《服务成果验收标准》为准。甲方认为不合格的，应在验收期内一次性书面列明不合格事项及依据；乙方完成整改后，甲方应在3个工作日内复验。甲方逾期未提出书面异议的，视为验收合格。”
- 对方不同意时：可接受将验收期限延长至10个工作日，但必须保留书面异议和逾期视为验收合格规则。
- 可接受底线/需客户判断：是否接受对方不设置“逾期视为验收合格”；如接受，需客户确认尾款回收风险。
- 可复制为批注的短文本：建议补充验收标准、验收期限、书面异议和逾期视为验收合格规则，否则尾款支付条件不清。
- 来源：合同原文第5条第2款；模型基于合同审查规则分析。
```
